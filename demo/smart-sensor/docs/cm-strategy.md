# Configuration Management Strategy — Smart Sensor

Per methodology §10.8. The CM strategy answers four questions:

1. **What is under configuration control.**
2. **How baselines are named and frozen.**
3. **How changes to baselined items are governed.**
4. **How traceability between baselines is preserved.**

## Items under control

The git tree is the configuration repository. Each commit on `main` is a configuration item by reference. The `.iso-config.yaml` `baselined_paths` list enumerates the paths that require an open Change Request to edit:

- `docs/project-plan.md`
- `docs/cm-strategy.md`
- `docs/risk-register.md`
- `methodology/`
- `model/library/`

Story files are not by default baselined. They become baselined when included in a `release-vN.M` baseline.

## Baseline naming

Two baseline tag families exist on `main`:

- `plan-baseline-vN.M` — Project Plan baseline, governing schedule and risk posture.
- `release-vN.M` — model release, freezing the set of `done` stories at the moment of release.

Both are annotated tags. The annotation contains the baseline scope (which CIs are frozen) and any supersession of an earlier baseline.

## Change governance

Edits to baselined paths require an open Change Request Issue with the `change-request` label. The commit-msg hook enforces a `CR #<n>` reference in the message body. The Change Request lifecycle (§10.4.2) is: submitted → evaluated → agreed → in implementation → done / rejected / postponed.

## Traceability

The traceability matrix is rendered from model relations (`derive`, `frame concern`, `verify`) by `tools/render/traceability-matrix.py`. The output lives at `docs/generated/traceability-matrix.md` and is regenerated by the post-merge hook. It is gitignored.

The matrix is regenerated at every release boundary and saved as a release artefact under `docs/releases/`.
