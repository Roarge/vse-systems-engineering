# SysML 2.0 Model Execution

Model execution, also known as simulation, is the use of a
machine to step or evolve a model so that its specified behaviour
can be observed. Executable models direct a simulator to perform
steps in a given order, or sets of differential equations that
can be solved or approximated, to obtain a sequence of simulated
system states (an **execution trace**). These traces can be
visualised in diagrams, replayed or observed in an interactive
environment, or reconstructed from a log file. Chapter 39 of the
SysML v2 book is new in the 2026-04 release (pages 284 to 289).

## Is SysML an executable modelling language?

The short answer is **no**. SysML 2.0 has always been a
specification language. It can capture abstract concepts about a
system from the very early stages of design. Direct executability
requires sufficient detail to decide what to do in the simulation,
preferably deterministically. In early phases, the design has so
many open questions that a deterministic simulation is typically
impossible, and in fact undesirable.

Rather than enabling direct executability, the semantics of KerML
and SysML define **what constitutes an acceptable execution**.
The model is not a recipe to generate an execution. It is a means
of checking whether an execution generated by an external source
(a simulator or an actual implementation) is correct with respect
to the model.

In the early stages of design, this check is very forgiving and
will accept many different behaviours. As the design proceeds and
more constraints are added, the set of valid behaviours is
narrowed, bringing the model closer to a single deterministic
execution trace. So although SysML 2.0 never generates an
execution, the gap between checking traces and producing them
shrinks as the design matures, which is what makes
execution-style analysis useful throughout the design process.

### Declarative versus operational semantics

This declarative stance is a deliberate departure from earlier
OMG approaches to executable modelling. fUML and PSSM both took
the **operational** path: they defined an execution engine (token
flows, dispatch order, step semantics), and a conforming tool was
expected to implement that engine. KerML and SysML 2.0 take the
**denotational** path: they define what counts as a valid
execution and leave the **how** to tools.

The benefit is **flexibility**: any execution mechanism
(discrete-event, continuous, constraint-solving, or a mix) can
claim conformance as long as the traces it produces satisfy the
model. The cost is that there is no recipe to follow, which is
exactly the gap that the rest of Chapter 39 discusses.

## What does it mean to execute a SysML model?

A comparison with a programming language can sharpen the point.
Consider the same intent expressed in Python and SysML 2.0:

```python
def func1():
    print("Function 1")

def func2():
    print("Function 2")

# Execute in order
func1()
func2()
```

```sysml
action def Func1;
action def Func2;

action def ExecuteFunctions {
    action f1 : Func1;
    action f2 : Func2;
    first start then f1;
    first f1 then f2;
}
```

Python conflates **declaring that something will happen** with
**making it happen**. Both occur at the call site in lines 8 and
9. SysML 2.0 separates them: line 5 declares structurally that a
`Func1` occurrence is composed within `ExecuteFunctions`, but does
not trigger anything on its own. Whether and when `f1` runs is
determined by the execution engine, with successions like
`first start then f1` and `first f1 then f2` constraining the
order. Source order in Python **is** the schedule. In SysML 2.0,
it is just textual layout, and the successions are what matter.

A second consequence: **the SysML 2.0 model never executes on its
own**. It defines the shape of any valid trace. Producing a
concrete trace is a separate step performed by an executor. This
is what makes SysML 2.0 a specification language rather than a
programming language.

In the vocabulary of [[sysml2-occurrences-4d]], an **execution
trace** is a complete 4D extent that satisfies all the constraints
of the model. Executing a SysML 2.0 model means producing one
such extent: a single specific complete life that is consistent
with everything the model asserts.

## How to execute SysML models?

Several execution paradigms can claim conformance to a SysML 2.0
model, and active tooling exists in each:

- **Discrete-event / step-based execution.** The model's behaviour
  is decomposed into discrete steps (actions, state transitions,
  and so on). The simulator picks the next step that is ready,
  executes it to advance the state, and repeats until the run
  ends. The fUML lineage. Well-suited to actions
  ([[sysml2-actions]]) and state machines
  ([[sysml2-state-machines]]).
- **Continuous / equation-based execution.** Solve or integrate
  differential equations over a time horizon. The SysPhS
  specification (OMG 2021a) extends SysML for this. SysPhS v2 is
  in progress. Well suited to parametric and physical models.
- **Constraint solving / SMT-based execution.** Treat the model
  as a logical formula and ask a solver to find a satisfying
  trace. NIST's recent work on verifying executability of SysML
  behaviour models via Alloy and SMT solvers fits here (Barbau
  and Bock 2020; Doerr et al. 2024).
- **Hybrid.** Most realistic systems need a mix. Cameo Simulation
  Toolkit and Ansys SAM combine discrete-event simulation with
  continuous solvers.

### Tool support

Two major tool vendors that participated in the work of the
Execution Working Group have already implemented prototype
simulators for SysML 2.0:

- **Ansys System Architecture Modeler (SAM)**.
- **Dassault Systèmes' Cameo Simulation Toolkit**.

Both offer simulation capabilities for SysML 2.0 models in line
with the proposed execution rules.

The work of the SysML v2 Submission Team (SST) has been picked up
by the Systems Modeling Community's Execution Working Group, but
has proceeded in a different direction. At the time of writing,
the Execution Working Group has compiled dozens of unit cases
(small models that capture different aspects of the language) to
analyse the semantics and compare them with user expectations.
Based on this work, they propose execution rules (heuristics for
selecting executions that provide the most value to users), leaning
into step-based execution. These artefacts are expected to form
Annex B of a future version of the SysML 2.0 specification (OMG
2025a).

## When to execute and what

In practice, the value of executing a model rises with the
maturity of the model. Very early designs are too underspecified
for execution to teach you anything that static analysis cannot
already reveal. Sampling traces from a still-vague set of
acceptable behaviours will mostly surface artefacts of arbitrary
tool choices rather than properties of the system.

Once architecture, behaviour, and key parameters are pinned down,
simulation becomes useful for scenario exploration, requirement
validation, and digital-twin synchronisation. **Variant
resolution** (see Chapter 35 and
[[sysml2-variant-configuration]]) is often a prerequisite, since
most executors expect a single configuration rather than a family
of them.

Execution engines typically require the user to select a model
element (preferably a definition) to instantiate. Typically the
system under consideration is not independent from its
environment, so a simulation must also mock a context. It is also
common to define the stimulus and monitor the system's reactions.
**Cases** (see [[sysml2-cases-overview]]) can describe exactly
this, making case definitions particularly clean entry points
for execution. They specify a subject (the system under
consideration), actors that configure the environment, and an
objective that says what the run is supposed to achieve. They can
also declare scenarios or define constraints for the execution,
and host `ToolExecution` or `ToolVariable` metadata annotations
to configure the tool.

Sometimes the case defines a simulation scenario which is executed
in different contexts. In such cases, part or occurrence
definitions defining the different contexts can wrap the case and
set its parameters (the subject and the actors). The entry point
for execution can then be the part or occurrence definition.
