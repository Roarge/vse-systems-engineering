# [Glossary Title]

A **glossary** page carries a set of term definitions scoped to a single
layer or concern. Use this type for term clusters that belong together
(SysML 2.0 behaviour vocabulary, ISO 29110 roles, PHAS-EAI constructs),
not for isolated single terms.

## When to use this type

- A concept page uses several terms that need precise definitions and
  pulling them into the concept page would dilute its explanatory flow.
- Multiple skills need to refer to the same term set consistently.
- The terms have canonical definitions in a source specification and the
  added value is curation, not original definition.

## What this page should contain

- Alphabetical or topical ordering of terms, either works but pick one
  and stick to it for the page.
- Each term: the term in bold, its definition, and a link to the source
  or to a `reference` page that expands on it where useful.
- Where the source defines a term ambiguously, choose one definition,
  state it, and note the alternative in a short parenthetical.

## What this page should not contain

- Original definitions. Glossary pages reflect source definitions, not
  plugin-specific coinages.
- Discussion or worked examples. Use a `concept` page for that.

## Size guidance

Glossary pages typically run 40 to 150 lines. A glossary over 200 lines is
usually mixing two subject areas and should be split.
